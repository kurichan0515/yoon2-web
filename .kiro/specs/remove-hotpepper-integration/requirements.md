# Requirements Document

## Project Description (Input)

### 仕様変更の概要
ホットペッパービューティーの連携が不可能となったため、それに関連する機能・コード・ドキュメントを削除し、よりシンプルな実装に調整する。

### 変更の背景
ホットペッパービューティーの情報を連携できないことが判明したため、以下の対応が必要：
- ホットペッパービューティー連携に関連するコードの削除
- 関連するドキュメントの削除または修正
- よりシンプルで直接的な予約管理システムへの移行

### 主な影響範囲
1. **calendarService.js** - ホットペッパー予約識別機能の削除
2. **ドキュメント** - 連携手順ガイドの削除または修正
3. **コメント** - コード内のホットペッパー関連コメントの削除

#### 1.2.2 予約ソース判定の簡素化
- **保持する予約ソース**:

  - その他
- **削除する予約ソース**:

#### 1.2.3 統計機能の簡素化
- **削除**: `hotPepperBookings` と `directBookings` の区別
- **変更後**: `totalBookings`（総予約数）のみ表示
- **保持**: サービス別、予約ソース別の内訳（ホットペッパー除く）

#### 1.2.4 カレンダー凡例の更新
- **削除**: ホットペッパービューティー予約の凡例項目

## 2. 非機能要件

### 2.1 コード品質
- **可読性**: ホットペッパー関連のコードを削除することで、コードの複雑性を軽減
- **保守性**: 将来的な機能追加が容易になるシンプルな構造を維持
- **一貫性**: 予約表示のロジックを統一し、一貫性のあるUIを実現

### 2.2 互換性
- **後方互換性**: 既存のGoogle Calendar連携機能は維持
- **データ互換性**: 既存の予約データ（Firestore）には影響なし
- **API互換性**: Google Calendar API の使用方法は変更なし

### 2.3 パフォーマンス
- **処理速度**: ホットペッパー判定ロジック削除により、イベント処理が若干高速化
- **レンダリング**: UIの簡素化により、カレンダーレンダリングが軽量化

### 2.4 ユーザー体験
- **UI一貫性**: 予約表示が統一され、視覚的にシンプルで理解しやすい
- **混乱の回避**: ホットペッパーとの連携を期待するユーザーへの誤解を防ぐ

## 3. 制約事項

### 3.1 技術的制約
- Google Calendar API の機能は維持する必要がある
- 既存のFirebase連携機能には影響を与えない
- React、React Big Calendar などの既存ライブラリは引き続き使用

### 3.2 ビジネス制約
- ホットペッパービューティーとの連携が技術的に不可能
- 予約管理の基本機能（カレンダー表示、予約詳細表示）は維持必須

### 3.3 時間的制約
- 既存機能への影響を最小限に抑えた迅速な実装が求められる
- 段階的なリリースではなく、一括での変更が望ましい

## 5. リスク管理

### 5.1 識別されたリスク

#### リスク1: 既存の予約データへの影響
- **発生確率**: 低
- **影響度**: 高
- **対策**: コード削除前にバックアップを取得、Firestore データには一切手を加えない

#### リスク2: 意図しない機能の破壊
- **発生確率**: 中
- **影響度**: 高
- **対策**: 段階的なコード削除、各ステップでの動作確認

#### リスク3: ドキュメントの不整合
- **発生確率**: 中
- **影響度**: 中
- **対策**: ドキュメント全体の見直し、矛盾がないかのチェック

- **発生確率**: 低
- **影響度**: 低
- **対策**: シンプルで明確なUI設計、必要に応じて変更内容の通知
